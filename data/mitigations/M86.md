# Description
Replace packaged or unsupported components when reuse would lead to long-term maintenance problems or when vendor infrastructure is hardly updated and maintained.

# Prerequisites
Assessment of documentation quality, vendor support, technical quality, and economic feasibility.

# Tools / Technologies
Packaged solution

# Trade-offs / Side Effects
Replacement cost and integration effort.

# Effectiveness
The paper gives ForeignAccount as an example where package replacement was selected to avoid long-term maintenance problems.

# Stakeholders
- Application architects; vendors; maintenance team; migration team

# Contextual Constraints

- Third-party packaged software with limited documentation or weak vendor support.


# Standardized Mitigation Entry
Package replacement mitigates maintenance risk when third-party legacy package functionality is poorly documented or no longer well supported.

# Key Quote for Prompting
> "So we decided to replace it with a packaged solution"

# Logic Pattern
IF a packaged component is poorly documented or unsupported THEN APPLY package replacement.

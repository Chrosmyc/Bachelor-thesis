# Description
The resulting microservices may not be sufficiently cohesive or loosely coupled if responsibilities, operations, and data ownership are not grouped correctly.

# Root Cause
The decomposition must split the monolith into small cohesive units and define physical limits for implementation, distribution, management, and monitoring.

# Trigger Event
Operations and data are grouped into microservices without clear responsibility, business capability alignment, or API-mediated data access.

# Impact
Tight service coupling; weak cohesion; increased maintenance effort; impaired independent development and deployment.

# Linked Risks
- R84

# Stakeholders

- Architects; developers; engineers

# Contextual Constraints

- Applies when identifying microservice candidates and defining their APIs and data access.


# Detection Source
Source Code / Architecture Diagram / Database Model / API Specification

# Standardized Risk Entry
If responsibilities and data access are not grouped carefully, decomposition can produce tightly coupled or weakly cohesive microservices.

# Key Quote for Prompting
> This allows a good decomposition that guarantees low coupling, as well as high cohesion.

# Logic Pattern
IF operations/data are split without clear responsibility and API-mediated ownership THEN risk of low cohesion or high coupling occurs.
